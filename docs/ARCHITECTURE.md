# Architecture

## Routing

Frostboard uses the Next.js App Router with a route group: `src/app/(dashboard)/`.
The parentheses mean the segment doesn't appear in the URL ‚Äî `/dashboard`,
`/customers`, etc. all share the same layout (sidebar + header) without it
showing up in the path. The root `src/app/page.tsx` simply redirects `/` to
`/dashboard`.

Each route follows a consistent pattern: `page.tsx` is a Server Component that
exports `metadata` (so each page gets its own `<title>`) and renders a
co-located client component (e.g. `CustomersView.tsx`) that holds all the
interactive state. This split exists because a `"use client"` page cannot
export `metadata` ‚Äî splitting the concerns keeps both.

## The chiselui / Next.js App Router boundary

chiselui ships its components without `"use client"` directives, even though
several use React hooks internally. Every file that imports anything from
`chiselui` must declare `"use client"` itself, or the build fails. This is
documented per-component as it was discovered, and no Server Component in this
project imports from `chiselui` directly ‚Äî all chiselui usage is isolated to
client components.

## Theming

chiselui's `ThemeToggle` owns its own theme state: it writes `data-theme` to
`<html>` and persists the choice to `localStorage` under the key
`chiselui-theme`. The root layout includes a small inline script that reads
that key and applies `data-theme` before the page paints, preventing a flash of
the wrong theme on reload (`suppressHydrationWarning` is set on `<html>` to
match).

Charts built with Recharts can't use `var(--token)` directly ‚Äî Recharts paints
raw SVG presentation attributes, where CSS custom properties don't resolve. The
`useChartColors` hook (`src/components/dashboard/use-chart-colors.ts`) reads the
computed token values from `document.documentElement` on mount and re-reads them
whenever `data-theme` changes (via a `MutationObserver`), so charts stay
correctly colored across theme switches.

## Mock data

All data lives in `src/lib/mock-data.ts` and is generated by a seeded
pseudo-random number generator (`mulberry32`) seeded with fixed constants, plus
a fixed reference date. This guarantees:

- Identical output between the server render and the client render (no
  hydration mismatches)
- Internally consistent figures ‚Äî KPIs like `totalMRR` are derived from the
  same customer records the tables display, not invented separately

## Toast notifications

See the README's "Design decisions" section ‚Äî chiselui's `ToastProvider` is not
SSR-safe, so Frostboard ships a drop-in replacement with the same API. Views
import it through `src/hooks/useToast.ts`, a thin re-export that keeps the import
path stable regardless of where the implementation lives.
